# JusticeAI 当前功能清单与工作流说明

本文档基于当前仓库代码实际状态整理，描述当前已经实现的功能、页面入口、后端接口、用户工作流与系统运行方式。

---

## 1. 项目当前定位

当前 JusticeAI 还处于「导入中心最小闭环」阶段。

现阶段重点不是完整 AI 审查流程，而是先把下面这条基础链路打通：

1. 用户打开浏览器界面
2. 先检查系统准备状态
3. 上传导入文件
4. 查看导入记录列表
5. 查看单条导入详情与关联文件

也就是说，当前项目更接近：

- 一个可在浏览器访问的前后端联调控制台
- 一个导入业务最小闭环原型
- 一个为后续接入 vLLM、HugeGraph、Milvus 等能力预留入口的运行骨架

---

## 2. 当前页面与入口

前端当前只有 3 个主要页面入口。

### 2.1 `/setup` 系统准备页

这是默认入口页，也是当前最重要的入口页面。

主要作用：

1. 检查后端是否在线
2. 检查健康接口是否可读
3. 展示当前运行配置
4. 展示 PostgreSQL / HugeGraph / vLLM / Milvus 的健康状态
5. 告诉用户如何手动启动系统
6. 告诉用户配置文件应该去哪里改
7. 在系统可用时，引导用户进入导入中心

当前页面上能看到的信息包括：

- 应用名
- 运行环境
- 后端监听地址
- 数据库连接池参数
- vLLM 地址和模型名
- HugeGraph 地址
- Milvus 地址
- 上传、报告、训练目录路径
- 依赖状态表

### 2.2 `/imports` 导入中心页

这是当前业务主页面。

主要能力：

1. 选择文件
2. 在前端进行文件合法性校验
3. 上传文件到后端
4. 查看导入记录列表
5. 切换分页
6. 按状态筛选
7. 查看导入详情
8. 查看关联文件列表
9. 上传成功后自动刷新列表并尽量定位到新记录

### 2.3 404 页面

如果用户访问了不存在的前端路由，会进入 404 页面。

404 页面当前提供两个回跳入口：

- 返回系统准备页
- 返回导入中心

## 3. 当前后端接口

当前后端已经接通的接口如下。

### 3.1 `GET /api/health`

作用：

- 返回后端健康状态
- 探测 PostgreSQL、HugeGraph、vLLM、Milvus 的状态

当前规则：

- PostgreSQL：会实际执行 `SELECT 1`
- HugeGraph：会探测 HTTP 地址
- vLLM：会探测 `${VLLM__BASE_URL}/models`
- Milvus：当前返回 `not_checked`

返回结果用于前端 `/setup` 页面展示。

### 3.2 `GET /api/system/info`

作用：

- 返回系统当前配置快照

返回内容包括：

- app 信息
- database 参数
- hugegraph 地址
- milvus 地址
- vllm 地址与模型名
- storage 目录
- runtime 版本

这个接口主要给 `/setup` 页面使用。

### 3.3 `POST /api/import/upload`

作用：

- 上传一个导入文件
- 写入磁盘
- 写入导入记录与导入文件记录

当前上传限制：

- 只允许：`csv`、`xls`、`xlsx`
- 单文件最大：**10 MB**
- 文件名不能为空
- 文件名长度不能过长
- 文件名中不能包含非法路径字符

上传成功后，后端会做两件事：

1. 把文件写到上传目录
2. 在 PostgreSQL 中写入：
   - `imports`
   - `import_files`

### 3.4 `GET /api/imports`

作用：

- 查询导入记录列表

支持参数：

- `page`
- `page_size`
- `status`

当前状态筛选只支持：

- `uploaded`

同时支持：

- 分页
- 总数返回
- 页码修正
- 页大小上限控制

### 3.5 `GET /api/imports/:id`

作用：

- 查询单条导入记录详情
- 查询这条导入记录下的关联文件

如果记录不存在，会返回 NotFound。

## 4. 当前数据库表结构用途

当前核心表有两张。

### 4.1 `imports`

表示一次导入任务或导入记录。

字段关注点：

- `id`
- `source_type`
- `status`
- `created_at`
- `updated_at`

### 4.2 `import_files`

表示一次导入关联的文件。

字段关注点：

- `id`
- `import_id`
- `original_filename`
- `stored_filename`
- `stored_path`
- `file_size`
- `mime_type`
- `created_at`

目前的业务关系是：

- 一条 `imports` 记录
- 对应一条或多条 `import_files`

## 5. 当前用户工作流

下面按真实用户操作顺序描述当前已经实现的工作流。

---

## 5.1 工作流一：系统准备检查

### 步骤 1：用户打开 Web UI

默认建议从：

- `http://127.0.0.1:18100`

进入。

### 步骤 2：进入 `/setup`

前端默认会引导用户先看系统准备页。

### 步骤 3：页面自动检查后端状态

页面加载时会自动请求：

- `/api/system/info`
- `/api/health`

### 步骤 4：前端展示结果

如果后端不可用：

- 显示「未连接」
- 提示用户先启动后端

如果后端可用但依赖不完整：

- 显示「部分可用」
- 允许用户带警告进入导入中心

如果依赖都可用：

- 显示「已连接」
- 可以正常进入导入中心

### 步骤 5：用户手动刷新状态

用户可以点击：

- 「重新检查后端状态」

重新发起检查。

---

## 5.2 工作流二：导入文件

### 步骤 1：进入 `/imports`

用户从系统准备页进入导入中心。

### 步骤 2：选择文件

用户通过文件输入框选择文件。

### 步骤 3：前端校验文件

前端会校验：

1. 是否选择了文件
2. 扩展名是否在允许范围内
3. 文件是否为空
4. 文件大小是否超过 10 MB

如果不通过：

- 直接在页面报错
- 不发送请求给后端

### 步骤 4：上传请求

前端以 `FormData` 方式调用：

- `POST /api/import/upload`

字段名为：

- `file`

### 步骤 5：后端处理上传

后端会：

1. 读取 multipart
2. 校验文件名与扩展名
3. 生成新的存储文件名
4. 按日期目录写入上传目录
5. 在数据库插入导入记录
6. 在数据库插入文件记录

### 步骤 6：前端展示上传成功

上传成功后页面会显示：

- 成功提示消息

同时会主动刷新列表。

---

## 5.3 工作流三：查看导入列表

### 步骤 1：页面加载时自动拉列表

导入中心在进入时会自动请求：

- `GET /api/imports`

默认参数：

- `page=1`
- `page_size=20`

### 步骤 2：列表展示导入记录

表格里当前会展示：

- 导入 ID
- 来源类型
- 状态
- 创建时间
- 更新时间
- 操作按钮

### 步骤 3：支持翻页

用户可以：

- 上一页
- 下一页

### 步骤 4：支持状态筛选

当前筛选只有一个实际业务状态：

- `uploaded`

### 步骤 5：支持刷新列表

用户点击「刷新列表」后，会再次请求导入列表。

---

## 5.4 工作流四：查看导入详情

### 步骤 1：用户点击「查看详情」

### 步骤 2：前端请求详情接口

调用：

- `GET /api/imports/:id`

### 步骤 3：前端展示详情区域

当前展示内容包括：

- 导入 ID
- 来源类型
- 状态
- 创建时间
- 更新时间

### 步骤 4：展示关联文件表

如果有文件，会展示：

- 原始文件名
- 存储文件名
- 相对存储路径
- 文件大小
- MIME 类型
- 创建时间

如果没有文件，会显示空提示。

---

## 5.5 工作流五：上传后自动定位新记录

这是当前体验上一个比较关键的细节。

上传成功后，前端会尝试：

1. 重新拉取第一页列表
2. 尝试在列表中找到刚刚新建的 `import_id`
3. 如果能找到，就自动加载这条记录详情
4. 如果列表暂时没定位到，也会提示用户并尝试加载详情

这让当前最小闭环体验更加连贯。

## 6. 当前系统依赖关系

目前系统运行依赖的角色如下。

### 6.1 硬依赖

#### PostgreSQL

这是当前后端启动的硬依赖。

原因：

- 后端启动时会直接建立数据库连接池
- 如果 PostgreSQL 不通，后端启动失败

### 6.2 软依赖 / 降级依赖

#### HugeGraph

当前主要用于健康检查展示。

#### vLLM

当前主要用于健康检查展示和系统配置展示。

#### Milvus

当前主要用于地址展示；健康检查尚未真正接通。

因此当前项目的真实情况是：

- **没有 PostgreSQL 不行**
- **没有 HugeGraph / vLLM / Milvus，系统仍可启动，但会显示降级**

## 7. 当前配置工作流

当前采用根目录统一配置入口。

### 7.1 本地开发配置

优先使用：

- `.env`

参考模板：

- `.env.example`

### 7.2 Docker 运行配置

参考模板：

- `.env.docker.example`
- `docker-compose.yml`

### 7.3 后端配置读取顺序

当前后端配置读取逻辑为：

1. 先读仓库根目录 `.env`
2. 再回退读 `backend-rust/.env`
3. 再读 `backend-rust/config/default.toml`
4. 再读环境对应的 `backend-rust/config/{APP_ENV}.toml`
5. 最后再叠加环境变量覆盖

## 9. 当前还没有实现的功能

这部分非常重要，因为它决定了目前项目「能做什么」和「还不能做什么」。

当前**还没有真正实现**的能力包括：

1. 没有真正的 AI 对话或推理调用流程
2. 没有真正把导入文件送进 vLLM 做分析
3. 没有案件解析、结构化抽取、规则判断、风险评分等核心业务逻辑
4. 没有任务队列、异步处理、批处理工作流
5. 没有用户系统、权限系统、登录系统
6. 没有正式报表生成流程
7. 没有把 HugeGraph / Milvus 真正接入业务查询链路
8. 没有模型管理页面
9. 没有导入后的二次处理页面
10. 没有审核流、审批流、结果回写流

所以当前项目的真实阶段应理解为：

- 浏览器 UI 壳已经有了
- 导入中心最小闭环已经有了
- 系统依赖与模型配置入口已经有了
- 但真正的 AI 核心业务链路还没接上

---

## 10. 当前最合理的下一步开发顺序

如果继续往下推进，建议顺序如下：

### 第一步：接通 vLLM OpenAI ChatCompletion 兼容调用

目标：

- 把当前只是展示配置的 vLLM，变成真正可以调用的模型服务

### 第二步：定义导入后的处理工作流

例如：

1. 上传文件
2. 解析文件内容
3. 切分文本或结构化数据
4. 发送给模型
5. 生成分析结果
6. 保存结果
7. 在前端展示处理状态和输出结果

### 第三步：补一个处理任务状态机

至少应有类似状态：

- uploaded
- parsing
- queued
- processing
- completed
- failed

### 第四步：给导入记录增加处理结果页

这样系统才会从「导入控制台」真正变成「业务分析系统」。

---

## 11. 一句话总结

当前 JusticeAI 已经实现的是：

**一个可在浏览器中访问、可通过 Docker 启动、具备系统准备检查 + 文件导入上传 + 导入列表 + 导入详情查看能力的最小闭环原型系统。**

但当前还没有真正完成：

**基于 vLLM 的 AI 推理业务链路。**
